# 自动化测试全流程插件原型 + React Flask 登录注册 Demo 任务分析与流程

## 1. 项目目标

本项目的目标是先完成一个最小可验证闭环，用一个可运行的 React + Flask 登录注册 Demo 系统承载完整测试流程，验证 AI 驱动自动化测试插件的核心链路是否成立。

最小闭环包括：

1. 输入 PRD 文档。
2. 从 PRD 中拆解结构化需求。
3. 基于需求生成测试点和测试用例。
4. 执行手工测试与接口测试。
5. 生成测试执行记录和测试报告。
6. 基于发现的问题提交标准化 Bug 单。
7. 沉淀插件原型架构与执行链路。

Demo 系统不追求业务复杂度，重点是让“需求到 Bug 单”的流程完整跑通。

## 2. Demo 系统范围

### 2.1 前端范围

前端使用 React、Axios、React Router、Ant Design，实现以下页面和流程：

1. 注册页面。
2. 登录页面。
3. 登录成功后的首页或用户信息展示页。
4. 基础路由跳转。
5. 表单输入、提交、成功提示、失败提示。

前端需要调用后端接口完成注册和登录，不使用纯前端模拟数据。

### 2.2 后端范围

后端使用 Flask、Python、SQLite，实现以下能力：

1. 用户注册接口。
2. 用户登录接口。
3. SQLite 用户表。
4. 密码存储处理。
5. 基础错误返回。
6. 跨域配置，支持前端本地调用。

### 2.3 预埋业务规则 Bug

需求规则：

用户名长度必须大于等于 6 位。

预埋缺陷：

后端注册逻辑不校验用户名长度，导致小于 6 位的用户名可以注册成功。

该缺陷用于测试流程验证：

1. PRD 中明确写出用户名长度规则。
2. 需求分析文档提取该规则。
3. 测试用例覆盖该规则。
4. 测试执行发现实际结果与期望结果不一致。
5. 测试报告记录该问题。
6. Bug 单标准化描述该问题。

## 5. 阶段拆解

### 阶段一：任务分析与交付边界确认

目标：

明确项目范围、系统边界、缺陷验证点、文档清单、实施顺序。

输出：

1. 任务分析与流程文档。
2. 后续目录规划。
3. 交付物清单。

完成标准：

项目后续工作可以按文档拆分执行，不需要再重新定义大方向。

### 阶段二：PRD 与需求规格文档

目标：

先让需求文本成为测试生成的输入源。

输出：

1. `docs/prd.md`
2. `docs/requirement-spec.md`

重点：

PRD 中必须明确写出“用户名长度必须大于等于 6 位”，为后续发现预埋缺陷提供依据。

### 阶段三：Demo 系统开发

目标：

开发一个可本地运行的登录注册系统。

后端任务：

1. 建立 Flask 应用。
2. 建立 SQLite 用户表。
3. 实现注册接口。
4. 实现登录接口。
5. 实现统一响应格式。
6. 在注册逻辑中保留用户名长度校验缺失问题。

前端任务：

1. 建立 React 工程。
2. 配置路由。
3. 实现注册表单。
4. 实现登录表单。
5. 使用 Axios 调用后端。
6. 使用 Ant Design 展示表单和提示。

完成标准：

1. 正常用户名可以注册。
2. 已注册账号可以登录。
3. 错误密码登录失败。
4. 小于 6 位用户名可以注册成功，用于触发预埋缺陷。

### 阶段四：测试用例设计

目标：

从需求规格中设计覆盖核心规则的测试用例。

输出：

1. `docs/test-cases.md`

用例范围：

1. 注册成功。
2. 用户名为空。
3. 用户名长度小于 6 位。
4. 密码为空。
5. 重复用户名注册。
6. 登录成功。
7. 密码错误。
8. 未注册用户登录。

重点用例：

用户名长度小于 6 位时，期望注册失败；实际系统注册成功。

### 阶段五：测试执行与记录

目标：

执行手工测试和接口测试，记录实际结果。

输出：

1. `docs/manual-test-execution.md`
2. `docs/api-test-execution.md`

执行方式：

1. 页面手工操作注册与登录流程。
2. 使用接口请求验证注册与登录接口。
3. 记录请求数据、期望结果、实际结果、执行状态。

完成标准：

缺陷用例必须能被复现，并保留可读的复现步骤。

### 阶段六：测试报告与 Bug 单

目标：

将执行结果转化为测试报告和标准 Bug 单。

输出：

1. `docs/test-report.md`
2. `docs/bug-report.md`

测试报告内容：

1. 测试范围。
2. 测试环境。
3. 测试用例数量。
4. 通过数量。
5. 失败数量。
6. 缺陷摘要。
7. 风险结论。

Bug 单内容：

1. 标题。
2. 严重程度。
3. 优先级。
4. 前置条件。
5. 复现步骤。
6. 期望结果。
7. 实际结果。
8. 影响范围。
9. 关联需求。
10. 关联测试用例。

### 阶段七：插件原型设计方案

目标：

把 Demo 中跑通的人工链路抽象成插件架构。

输出：

1. `docs/plugin-prototype-design.md`

插件链路：

```text
PRD 文本
  -> PRD 解析
  -> 结构化需求
  -> 测试点提取
  -> 测试用例生成
  -> 接口或页面执行
  -> 执行结果归档
  -> 测试报告生成
  -> Bug 单生成
  -> Bug 系统推送
```

每个节点需要定义：

1. 输入。
2. 输出。
3. 处理逻辑。
4. 可落地的最小实现。
5. 后续扩展方向。

## 6. 验收标准

项目完成时应满足以下标准：

1. 前端和后端可以本地启动。
2. 注册和登录基础流程可正常操作。
3. 预埋缺陷可以稳定复现。
4. PRD、需求规格、测试用例、执行记录、测试报告、Bug 单齐全。
5. 文档之间存在清晰追踪关系：需求规则 -> 测试用例 -> 执行失败 -> Bug 单。
6. 插件原型方案能说明后续如何从人工文档流程升级为自动化插件流程。
7. 交付物可以用于简历描述，体现开发、测试、AI 流程设计的完整闭环。

## 7. 简历复用表达方向

该项目可在简历中突出以下能力：

1. 从零设计并实现 React + Flask 前后端分离 Demo 系统。

3. 构造业务规则缺陷并完成完整质量验证闭环。
4. 设计 AI 自动化测试插件最小原型，覆盖 PRD 解析、用例生成、自动执行、报告生成、Bug 推送。

建议后续按以下顺序推进：

1. 先补齐 `docs/prd.md` 和 `docs/requirement-spec.md`。
2. 再开发 Flask 后端，保证接口行为可测。
